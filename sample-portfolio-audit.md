# 脱敏样例：AI 项目作品集体检

审查对象：消费级 AI 沟通助手（公开作品集）  
审查范围：GitHub README 与项目叙事  
说明：这是服务格式样例，不代表对客户材料的真实评价。

## 30 秒第一印象

**结论：方向有辨识度，但需要更快证明“这是可运行的产品，不是概念图”。**

招聘方能理解产品用途，也能看到安全意识；但如果首屏先展示长段背景，关键证据会被淹没。建议把“可运行闭环、真实接口、明确边界”提到最前面。

## 三个最大问题

### P0 — 首屏缺少可验证证据

目前的产品定位清楚，但首屏没有马上回答：能不能跑、用户能做什么、结果长什么样。

**建议：** 在标题下加入 3 个证据点：可运行交互、结构化 API、明确的人类确认边界，并放一个 60 秒 demo 路径。

### P1 — “AI”与本地规则引擎的边界出现太晚

如果项目当前使用确定性本地分析，而不是外部模型调用，这不是缺点；但必须尽早说明，否则招聘方会怀疑演示是否夸大。

**建议：** 新增 `What is real / What is mocked`，分别列出真实路由、交互状态、本地存储，以及尚未接入的生产模型、账号同步和提醒服务。

### P1 — 缺少产品判断的压缩表达

页面描述了许多功能，却没有一句话说明为什么流程是“情绪识别 → 用户目标 → 多语气草稿 → 人类确认”。

**建议：** 把核心循环写成 4 步，并解释关键原则：AI 只提供选项，用户保留发送权。

## 可直接替换的项目介绍

> MoodPilot 是一个面向高压沟通场景的消费级 AI 产品原型。用户粘贴一条难回复的消息后，系统先识别可能的情绪信号和用户目标，再生成多种语气的回复草稿；所有复制、保存和后续动作都需要用户确认。当前版本包含可运行的结构化分析接口和完整交互闭环，并明确披露本地规则引擎、模拟提醒和本地存储等演示边界。

## 24 小时修改顺序

1. 重写 README 首屏：一句话定位 + 3 个证据点 + demo 路径。
2. 增加 `What is real / What is mocked`，消除可信度疑问。
3. 把核心循环压缩为 4–6 步，突出用户控制权。
4. 补一张真实界面截图或 30–60 秒录屏。
5. 最后再补技术栈；不要让技术栈抢在用户价值之前。

## 交付边界

- 本次体检不包含完整 UI 重做或代码开发。
- 只基于已提供、可验证的信息改写，不新增未经证实的数据。
- 客户可在交付后 48 小时内提出 1 次小范围修改。
